# План: бесплатный генератор сопроводительных писем (SEO + воронка)

> Статус: **план, не реализовано.** Цель — отдельная посадочная с бесплатным
> ИИ-генератором сопроводительного письма под ключевые запросы «сопроводительное
> письмо нейросеть/ИИ/генератор/онлайн», которая (1) приводит органику и ссылки и
> (2) заводит людей в продукт. Главный нюанс — **жёстко ограничить бесплатность**,
> потому что каждый вызов модели стоит денег.

---

## 1. Зачем это (роль в SEO и воронке)

- **SEO-кластер с низкой конкуренцией.** «Сопроводительное письмо …» — частотные
  запросы; связка с «нейросеть/ИИ/генератор» слабо занята, в отличие от «вакансии».
- **Бесплатный инструмент = linkable asset.** Его шарят и линкуют (форумы, Telegram,
  VK, подборки сервисов) — это даёт бэклинки, которых молодому домену не хватает.
- **Поведенческие сигналы** (особенно важно для Яндекса): тулом реально пользуются.
- **Идеальный интент → конверсия.** Кто ищет «сгенерировать письмо», прямо сейчас
  откликается. Бесплатный тул — крючок, регистрация + полный продукт (письмо под
  конкретную вакансию с подбором и оценкой совпадения) — апселл.

## 3. URL и структура страницы

- **URL (canonical):** `/soprovoditelnoe-pismo` — ключ прямо в пути.
  (Альтернатива `/generator-soprovoditelnogo-pisma`; выбрать один, второй → 301.)
- **Серверный компонент** (SSR, контент индексируется), внутри — лёгкий клиентский
  виджет-тул:
  1. `<h1>` + интро под ключ.
  2. **Тул** (client component): два поля — «Вакансия» (должность/описание) и
     «О себе / резюме» → кнопка «Сгенерировать письмо» → результат + «Скопировать».
  3. Под результатом — мягкий CTA: «Письмо под конкретную вакансию с подбором и
     оценкой совпадения — в личном кабинете» → `/profile`.
  4. **SEO-контент** (серверно): что такое сопроводительное письмо, как написать
     (пошагово = `HowTo`), 2–3 примера под разные роли, FAQ.
- **JSON-LD:** `WebApplication` (тул) + `HowTo` (как написать) + `FAQPage` +
  `BreadcrumbList`.
- **Discovery:** ссылка в футере; внутренние ссылки со страниц `/vakansii/*`
  («сгенерировать сопроводительное письмо»); запись в `sitemap.ts`.

## 4. Бесплатный эндпоинт (ungated, но защищённый)

Новый роут `POST /api/free/cover-letter` — **отдельный** от авторизованного, со
всей защитой ниже. Возвращает `{ text }` или `{ error, code }` (`code` =
`rate_limited` | `global_cap` | `too_long` | `bad_request`), чтобы фронт показал
правильный CTA.

## 5. ⭐ Как ограничить бесплатность (ядро плана)

Многослойная защита — от самой дешёвой к более сложной. Каждый слой режет и
стоимость, и абьюз.

### Слой 0. Жёсткие лимиты на сам запрос (бесплатно, ставим сразу)
- **Размер входа:** резюме ≤ 3000 симв., вакансия ≤ 3000 симв. Больше — `too_long`
  (или молча усечь). Не даём раздувать промпт.
- **Размер выхода:** `maxTokens` = **500** (против 900 у авторизованного). Бесплатное
  письмо короче — и дешевле, и оставляет ценность для платного.
- **Дешёвая модель:** для free режима — самая дешёвая приемлемая модель через
  параметр `model` у `chatCompletion` (он уже поддерживается). Дорогую модель — только
  авторизованным.

### Слой 1. Рейт-лимит по «личности» (главный контроль расхода)
Аноним идентифицируется **связкой**, чтобы пережить чистку кук:
- **IP** (из `x-forwarded-for` / `x-real-ip`, на Vercel есть) — храним **хэш** IP
  (HMAC с секретом), не сырой IP (152-ФЗ / приватность).
- **Кука** (httpOnly, signed) — выдаём при первом заходе.

Лимиты (стартовые, занижаем при росте затрат):
- **3 письма / сутки на IP**, **2 / сутки на куку**.
- **Burst-лимит:** не чаще **1 раза в 30 сек** на IP (стоп быстрому скриптингу).

Хранилище — таблица `free_tool_usage` + атомарный инкремент через RPC (§6).

### Слой 2. Глобальный «рубильник» расхода (потолок на день)
Глобальный дневной счётчик всех бесплатных генераций. Превысил бюджет (напр.
**500/сутки**) → free-тул отвечает `global_cap`: «Лимит на сегодня исчерпан, зайдите
завтра или зарегистрируйтесь». **Ограничивает максимальный дневной расход** при любой
атаке — это нижняя страховка.

### Слой 3. Анти-бот хардненинг
- **Honeypot** — скрытое поле; заполнено ботом → тихо отклоняем.
- **Time-to-submit** — время от рендера формы до сабмита < ~2 сек → подозрительно.
- **Origin/Referer** — принимаем только со своего домена.
- **Опционально (включить при появлении абьюза):** невидимая
  **Yandex SmartCaptcha** или **Cloudflare Turnstile** (бесплатные). Заложить хук,
  включить позже — не портим конверсию заранее.

### Слой 4. Кэш одинаковых запросов
Хэш(резюме+вакансия) → если такой запрос уже был недавно, возвращаем сохранённый
результат **без нового вызова модели**. Гасит абьюз через refresh/повторы.

### Слой 5. UX-нудж вместо наказания
- Бесплатное письмо «хорошее, но базовое»: короче, без подгонки под конкретную
  вакансию с оценкой совпадения, без сохранения/нескольких версий. Полная ценность —
  по регистрации (там и так бесплатный тариф щедрее: `FREE_LIMITS` = 10 откликов,
  100 анализов — см. [lib/plans.ts](src/lib/plans.ts)).
- **Email-gate как средний шаг (опция):** 1–2 письма бесплатно, затем «введите email,
  чтобы продолжить» — и режет расход, и собирает лиды до полной регистрации.

### Рекомендованные стартовые цифры (все легко тюнятся)
| Параметр | Значение |
|---|---|
| На IP / сутки | 3 |
| На куку / сутки | 2 |
| Burst | 1 / 30 сек |
| Выход `maxTokens` | 500 |
| Вход (резюме / вакансия) | 3000 / 3000 симв. |
| Глобальный потолок / сутки | 500 писем |
| Модель | самая дешёвая приемлемая |

**Грубая оценка расхода при потолке:** 500 писем/сутки × (~1500 вход + ~400 выход
токенов) — порядок копеек-рублей в день на дешёвой модели. То есть верхняя граница
дневного «слива» предсказуема и мала; реальная цифра — после выбора модели.

## 6. Модель данных (Supabase)

Таблица счётчиков (применить в SQL Editor, как и `app_tokens`):

```sql
create table if not exists public.free_tool_usage (
  bucket text primary key,           -- "ip:<hash>:2026-06-12" | "cookie:<id>:..." | "global:2026-06-12" | "ip:<hash>:burst:<window>"
  count integer not null default 0,
  updated_at timestamptz not null default now()
);
-- пишет только service-role (сервер), RLS закрыт для всех остальных
alter table public.free_tool_usage enable row level security;
```

Атомарная проверка-и-инкремент (без гонок), инкремент только если под лимитом:

```sql
create or replace function public.bump_free_usage(p_bucket text, p_limit int)
returns boolean language plpgsql as $$
declare v_count int;
begin
  insert into public.free_tool_usage (bucket, count) values (p_bucket, 1)
  on conflict (bucket) do update
    set count = free_tool_usage.count + 1, updated_at = now()
    where free_tool_usage.count < p_limit
  returning count into v_count;
  return v_count is not null;   -- null = лимит достигнут, не инкрементнули
end; $$;
```

Сервер вызывает `getSupabase().rpc("bump_free_usage", { p_bucket, p_limit })` для
каждого бакета (IP/сутки, кука/сутки, burst, глобальный). Разрешаем генерацию,
только если **все** вернули `true`. Старые суточные строки чистит периодический
прогон (или просто копятся — они крошечные).

## 7. Приватность (152-ФЗ)
- IP — только в виде HMAC-хэша, сырой не храним.
- В оферте/политике упомянуть, что для защиты от злоупотреблений считаем
  обезличенные обращения к инструменту.

## 8. SEO-специфика
- `metadata` (title/description/canonical), OG/Twitter уже глобально через файловую
  конвенцию.
- JSON-LD: `WebApplication` + `HowTo` + `FAQPage` + `BreadcrumbList`.
- Контент содержательный (примеры писем под роли, гайд «как написать») — не тонкая
  страница ради ключа.
- Перелинковка: футер, `/vakansii/*` → тул, тул → `/vakansii` и `/profile`.
- Добавить URL в `sitemap.ts`.

## 12. Фазы и оценка
- **Фаза A — ядро (~half-day–day):** страница + клиентский тул + hardened
  `/api/free/cover-letter` (инпут-капы, рейт-лимит IP+кука, глобальный рубильник,
  honeypot) + таблица/RPC в Supabase + SEO-контент + JSON-LD + sitemap + перелинковка
  + закрыть дыру в `/api/cover-letter` (§10).
- **Фаза B — по необходимости:** SmartCaptcha/Turnstile, кэш одинаковых запросов,
  email-gate, программные варианты под профессии.

## 13. Метрики успеха / мониторинг
- Дневное число бесплатных генераций и расход в кабинете AITunnel (не выше потолка).
- Конверсия `free_letter_to_signup` в Метрике.
- Позиции/показы по «сопроводительное письмо …» в GSC/Вебмастере.
- Если расход растёт — занижаем лимиты §5; если конверсия низкая — усиливаем CTA.
